# Security Scanner CI Preflight

Issue #807 adds Trivy and OSV-scanner as advisory CI jobs. This note records
the implementation boundaries so the scanners extend the existing CI quality
surface without creating a second policy or security-governance model.

## Architecture Boundaries

- Keep both scanners in `.github/workflows/ci.yml` as CI jobs, not application
  runtime features. They must not introduce controllers, DTOs, database tables,
  Ground Control requirements, traceability schemas, exception classes, or
  backend service abstractions.
- Preserve the existing CI shape: pinned third-party actions, least-privilege
  job permissions, explicit artifacts, and repo-native guardrails through
  `make policy`.
- Treat Trivy and OSV-scanner as advisory evidence producers for this issue.
  The configured scanner commands should use non-blocking exit behavior and
  upload SARIF or JSON artifacts for review.
- Keep SonarCloud and gitleaks as distinct existing controls. Trivy adds
  container, Dockerfile, secret, misconfiguration, and IaC coverage; OSV-scanner
  adds Maven/Gradle dependency vulnerability coverage. Do not merge their
  semantics into a single "security scan" abstraction.
- Scope Trivy narrowly to the issue contract: `backend/Dockerfile` image
  scanning and `deploy/docker/` IaC manifests. If the implementation builds an
  image for scanning, it should reuse the existing Dockerfile and repository
  context rather than creating a scanner-specific Docker build path.
- Scope OSV-scanner to Java / Gradle dependency inputs. Gradle dependency locks
  are opt-in in this repo; enable Gradle dependency locking first and commit the
  generated lockfile artifacts before wiring OSV-scanner against them.

## Cross-Cutting Concerns to Reuse

- **CI workflow conventions:** follow `.github/workflows/ci.yml`'s job style,
  action pinning, artifact upload pattern, Java setup, Gradle cache setup, and
  comments for intentional gating behavior.
- **Gradle ownership:** dependency metadata belongs under `backend/` and should
  be generated by Gradle's dependency-locking support, not hand-maintained by a
  custom parser or scanner script.
- **Policy gate:** run `make policy` before declaring the implementation
  complete. If CI or policy workflow surfaces change materially, keep
  `docs/DEVELOPMENT_WORKFLOW.md`, `docs/WORKFLOW.md`, and applicable ADR policy
  surfaces synchronized.
- **Deployment boundary:** deployment remains GHCR image publish plus smoke test
  plus tailnet-only deploy per ADR-030. Advisory scanner results must not alter
  deploy authentication, runner selection, or production environment variables.

## Guardrails

- Do not add a new repo-local vulnerability database, suppression schema, or
  scanner result persistence model. GitHub Actions artifacts are the storage
  boundary for this advisory change.
- Do not fail CI on findings unless a later issue explicitly changes scanner
  policy. Use scanner exit-code controls and `continue-on-error` only where the
  tool's exit behavior cannot fully express advisory mode.
- Do not duplicate gitleaks by making Trivy's secret scan the authoritative
  secret gate. Gitleaks remains the dedicated secret control; Trivy secret
  output is additional advisory evidence.
- Do not broaden scans to the whole repository without an explicit contract
  change. Wider scans can add noisy findings and make the advisory jobs harder
  to reason about.
- Do not upload build outputs, dependency caches, container layers, secrets, or
  environment dumps as artifacts. Upload only SARIF or JSON scanner reports.
- Do not use self-hosted runners for these jobs. They do not need tailnet
  access; use `ubuntu-latest` like the existing build/test/security-adjacent CI
  jobs.

## Non-Goals

- No policy decision that scanner findings block PRs, image publication, smoke
  tests, or deployment.
- No replacement for SonarCloud, gitleaks, Checkstyle, SpotBugs, Error Prone,
  OpenJML, or JaCoCo.
- No application-level security feature, endpoint, dashboard, database model, or
  Ground Control control-pack integration.
- No centralized vulnerability triage workflow or suppression lifecycle beyond
  what the scanners and GitHub Actions artifacts already provide.
