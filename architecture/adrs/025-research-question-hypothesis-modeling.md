# ADR-025: Research Question and Hypothesis Modeling via Requirement Conventions

## Status

Accepted

## Date

2026-05-05

## Context

ADR-024 chose to layer the research workflow over existing primitives
rather than create a parallel research subsystem. That decision implies a
modeling question: how do *research questions* and *hypotheses* live in
the existing requirements graph alongside software requirements?

Three options were considered:

1. **New `RequirementType` enum values** (`RESEARCH_QUESTION`,
   `HYPOTHESIS`). Touches the Java enum, every place the enum is mirrored
   (DTO, MCP enum contract, frontend mirror, tests, migrations), and is
   forever — enums are hard to remove. Per ADR-J002 and the
   `architecture-model-artifacts` note, this is reserved for cases where
   the existing taxonomy is *materially insufficient*.
2. **New entity type.** Rejected by ADR-024.
3. **UID + relation conventions on existing Requirements.** Add no schema;
   carve out a UID namespace and use `REFINES` / `RELATED` to express
   research-specific relations.

## Decision

Use option 3: research questions and hypotheses are stored as
**Requirements with project-scoped UID prefixes and prescribed
relations**.

### UID convention

For a Ground Control project with identifier `<proj>`:

| Concept | UID pattern | Example |
|---|---|---|
| Research question | `<PROJ>-RQ###` | `APTL-RQ001` |
| Hypothesis (refines a question) | `<PROJ>-H###` | `APTL-H001` |
| Follow-up question generated by Publication | `<PROJ>-RQ###` | `APTL-RQ014` |

Where `<PROJ>` is the upper-cased Ground Control project identifier with
hyphens removed. Numbers are sequential, never reused, and never reset
across waves. The `RQ` and `H` UID forms must match
`^[A-Z][A-Z0-9]*-(RQ|H)\d{3,}$`.

The `/research` skill and the new MCP helpers are the only sanctioned
authoring paths. Both validate the UID against this regex before filing
the Requirement.

### Requirement field conventions

Research questions and hypotheses are filed as Requirements with:

- `requirement_type`: `FUNCTIONAL` (default) or `NON_FUNCTIONAL` for
  observational research that cannot be falsified by a single experiment.
  No new enum values.
- `priority`: MoSCoW value reflecting research priority within the wave.
- `wave`: research wave (independent of software waves; operators may
  choose to keep them numerically separated).
- `statement`: full charter / hypothesis statement.
- `rationale`: why this question, what gap it fills.
- `status`: standard lifecycle — `DRAFT` while charter is being authored,
  `ACTIVE` once methodology is accepted, `DEPRECATED` if superseded by a
  follow-up, `ARCHIVED` once published and no further work expected.

### Relation conventions

| From | Relation | To | Meaning |
|---|---|---|---|
| Hypothesis | `REFINES` | Research question | Hypothesis is a concrete, testable instance of the question. |
| Research question | `DEPENDS_ON` | Research question | Cannot be answered until the dependency is. |
| Follow-up question | `RELATED` | Originating question | Surfaced during synthesis, recorded by Publication phase. |
| Superseding question | `SUPERSEDES` | Old question | Used when scope changes mid-flight. |

### Traceability conventions

| Artifact | `artifact_type` | `link_type` |
|---|---|---|
| Lit review document | `DOCUMENTATION` | `DOCUMENTS` |
| Methodology ADR | `ADR` | `DOCUMENTS` |
| Protocol document / spec | `SPEC` | `DOCUMENTS` (if narrative) or `VERIFIES` (if executable check) |
| Captured artifact (log, pcap, notebook) | `DOCUMENTATION` | `DOCUMENTS` |
| Verification step result | `PROOF` or `TEST` | `VERIFIES` or `TESTS` |
| Synthesis document | `DOCUMENTATION` | `DOCUMENTS` |

These rows reuse the existing `ArtifactType` and `LinkType` enums verbatim.

### Lab assets

Operational lab assets (hosts, networks, ranges, identities) are filed as
existing `Asset` records using the existing `AssetType` enum. Topology
is expressed via existing asset relations
(`CONTAINS`, `COMMUNICATES_WITH`, `TRUST_BOUNDARY`, etc.). Asset →
research-question linkage uses Asset Links with
`target_type=REQUIREMENT`.

### Project scoping

Every research question Requirement is scoped to the active Ground
Control project resolved from the repo's `AGENTS.md`
(`gc_get_repo_ground_control_context`). This matches the
`/implement` workflow.

## Consequences

### Positive

- Zero new schema, zero new enum values, zero new migrations.
- Research questions inherit the full requirements toolset for free:
  status transitions, audit trail, similarity search, similarity
  embedding, cycle detection, baselines, timeline export.
- The `gc_export_requirements` matrix and the existing dashboard count
  research questions automatically.
- Filtering existing UI by UID prefix or by the regex above reliably
  isolates the research view when needed.

### Negative

- Filtering / counting research-only views must be done at query time
  (UID prefix). The platform has no first-class field that says
  "this is a research question". A future requirement may add a
  metadata field if this proves limiting.
- `requirement_type` is reused with semantics the operator must learn.
  This is documented in
  `docs/research/RESEARCH_WORKFLOW_REQUIREMENTS.md`.

### Risks

- **Free-text drift.** Operators authoring requirements outside the
  `/research` skill could pick UID forms that violate the regex.
  Mitigation: the MCP helpers expose a `validateResearchUid` function
  used by the skill; the requirements doc records the convention; future
  work can add a quality gate enforcing it.
- **Numbering races.** Two parallel `/research` runs could pick the same
  UID. Mitigation: the skill resolves the next UID immediately before
  filing the Requirement, using existing list APIs; collisions surface
  as a normal Requirement-create conflict and are retried.
