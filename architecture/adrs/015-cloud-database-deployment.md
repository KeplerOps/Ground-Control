# ADR-015: Cloud Database Deployment

## Status

Accepted

## Date

2026-03-12

## Context

Ground Control's requirements data currently lives in a local Docker volume (PostgreSQL via Docker Compose). This data is the source of truth for the system's requirements graph — it includes requirements, relations, traceability links, GitHub issue sync records, and Envers audit history.

Local Docker volumes are ephemeral. A `docker compose down -v`, a machine rebuild, or a disk failure destroys the data. For a system whose purpose is requirements traceability, this is unacceptable.

We need durable storage while maintaining fast local development. The application itself continues running locally — only the database moves to the cloud.

### Relationship to ADR-002

ADR-002 established PostgreSQL as the primary database. This ADR extends that decision to cloud deployment via AWS RDS.

### Relationship to ADR-005

ADR-005 documented that Apache AGE "cannot use managed PostgreSQL services (RDS, Cloud SQL) without losing graph features." This trade-off is explicitly acceptable: all graph analysis works via JPA (`AnalysisService`, `GraphAlgorithms`), and AGE is optional (`groundcontrol.age.enabled=false` is the default). The AGE Docker image remains available for local development if graph queries are desired.

## Decision

Deploy PostgreSQL 16 on AWS RDS in the `catalyst-dev` account (516608939870), `us-east-2` region.

### Instance specification

- **Instance class:** `db.t4g.micro` (2 vCPU, 1 GiB RAM) — smallest viable for pre-alpha single-user
- **Storage:** 20 GiB gp3
- **Engine:** PostgreSQL 16
- **Multi-AZ:** No (pre-alpha, single user)

### Security model

- **Network:** Publicly accessible with security group restricting inbound to developer's IP (`/32`) on port 5432. No VPN or private subnet — proportional to threat model (single developer, known IP, no sensitive customer data).
- **Encryption in transit:** TLS required (`rds.force_ssl=1`, connection string `sslmode=require`)
- **Encryption at rest:** RDS storage encryption (AES-256, AWS-managed KMS key)
- **Credentials:** Random password generated by Terraform, stored in SSM Parameter Store (SecureString). Retrieved via `aws ssm get-parameter` for local development.
- **Application auth:** None. The API runs on localhost only. Application-level authentication will be added when the application itself moves to the cloud.

### Key sub-decisions

| Decision | Rationale |
|----------|-----------|
| **RDS over Aurora** | Aurora is overkill for single-user pre-alpha. `db.t4g.micro` is ~$12/mo vs Aurora minimum ~$30/mo. |
| **No AGE on RDS** | Not available as RDS extension. Acceptable per ADR-005 — all analysis works via JPA. |
| **No Redis** | Redis is configured but unused (`spring.cache.type=none`). Not deployed. |
| **No VPN/private subnet** | Proportional to threat model. Single developer, known IP, no customer data. SG + TLS is sufficient. |
| **SSM over Secrets Manager** | SSM Parameter Store SecureString is free. Secrets Manager is $0.40/secret/month. No rotation needed at this stage. |
| **Public accessibility** | Required for local app → cloud DB connectivity without VPN. Secured by SG + TLS + strong password. |

### Infrastructure as code

All resources defined in Terraform under `deploy/terraform/`. Structure:

```
deploy/terraform/
  bootstrap/          # S3 state bucket + DynamoDB lock (one-time manual apply)
  modules/
    networking/       # Security group, IP management
    rds/              # RDS instance, parameter group, subnet group
    secrets/          # SSM parameters for credentials
  environments/
    dev/              # Environment-specific wiring, backend config
```

Terraform state stored in S3 with DynamoDB locking, encrypted, versioned.

### Terraform CI/CD

- **Plan on PR:** `terraform fmt -check`, `validate`, and `plan` run on every pull request touching `deploy/terraform/**`. Plan output posted as PR comment for review.
- **Apply via `workflow_dispatch` only:** No auto-apply on merge. Infrastructure changes require explicit manual trigger with `action=apply`.
- **AWS auth via OIDC federation:** GitHub Actions assumes an IAM role via OpenID Connect — no long-lived AWS access keys stored as secrets.
- **OIDC provider + IAM role created by bootstrap Terraform:** The `deploy/terraform/bootstrap/` configuration creates the OIDC identity provider and IAM role that CI uses.
- **Bootstrap exception:** The bootstrap configuration itself is applied manually (it creates the state backend that all other Terraform depends on).

## Consequences

### Positive

- Data survives machine loss, rebuilds, and Docker volume cleanup
- Automated daily backups with 7-day retention
- Encryption at rest and in transit
- Infrastructure reproducible from code
- Plan visibility in PRs — reviewers see exact infrastructure changes before merge
- No long-lived AWS credentials — OIDC federation eliminates secret rotation burden
- Cost: ~$12-15/month (acceptable for pre-alpha)

### Negative

- Network latency: ~20-50ms to us-east-2 (vs <1ms local Docker). Acceptable for development workload.
- Developer IP management overhead: security group must be updated when IP changes
- No AGE graph queries on cloud DB (acceptable — JPA analysis is the primary path)
- Terraform state management overhead
- OIDC provider + IAM role add bootstrap complexity (one-time manual setup)

### Risks

- **IP changes break connectivity.** Mitigated by Makefile target (`make cloud-db-ip`) to update security group.
- **Terraform state contains sensitive values.** Mitigated by encrypted S3 bucket with versioning and access controls.
- **RDS cost if left running.** ~$12-15/mo is acceptable for pre-alpha. Can stop instance during extended breaks.
- **Latency affects development experience.** Monitor and consider moving to a closer region if problematic.
